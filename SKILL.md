# 记忆压缩与强制检索协议

## 触发词
- "压缩/整理/瘦身记忆"
- "MEMORY太大了"
- "上下文太长了"
- "记忆管理"
- "compress memory"

## 核心工作流 (IF 触发 THEN 执行)

1. **探测与评估 (多维健康度判定)**
   - 扫描根目录寻找核心记忆文件 (如 `MEMORY.md`, `.cursorrules`, `CLAUDE.md`)
   - **健康判定** (满足以下任一条件即触发压缩)：
     - 行数阈值：`>150行` (经验基准线，防止人类阅读疲劳)
     - Token估算：`>4000 Tokens` (约10KB)，防止"Lost in the Middle"效应
     - 结构恶化：文件中出现大量冗长代码块或超过3级的历史Bug记录
   - 若健康：报告"记忆文件信噪比良好，无需压缩"
   - 若触发：自动启动压缩流程

2. **基建 (自动创建)**
   - 在项目根目录创建 `docs/` 文件夹（若不存在）
   - 创建/更新 `docs/00-Index.md` 作为双链索引

3. **转移 (先写)**
   - 提取记忆中的 [技术细节/代码/历史Bug/配置]
   - 按模块写入 `docs/[Module]-Details.md`
   - 子文档注入：顶部 `← [[00-Index]]`，底部 `#project-archive`
   - 更新 `00-Index.md` 追加新链接

4. **精简 (后删)**
   - 从原记忆文件删除已转移细节，仅保留 [项目简介/核心架构/当前任务/红线]
   - **⚠️ 关键动作**：在删除处注入"强钩子"（见下方格式）

5. **验证与汇报**
   - 确认原文件通过多维健康判定 (行数<150, Token<4000, 结构清晰)
   - 确认 `docs/` 文件已落盘
   - 汇报："✅ 记忆压缩完成：[文件名] 从 [X]行→[Y]行 (减少 Z%)，详情已归档至 docs/"

## ⚠️ 强钩子格式规范 (写入核心记忆文件)

禁止使用弱链接 `详见 [[xxx]]`，必须使用以下强指令格式：

> `🛑 涉及 [模块名] 开发/修改前，【必须调用 read_file 工具】读取 [[docs/xxx-Details.md]] 获取最新细节，严禁凭上下文脑补！`

## 🛡️ 强制检索协议 (Agent 日常开发必读)

当本 Skill 被加载，或 Agent 在处理后续开发任务时，必须遵守以下铁律：

- **触发条件**：任务涉及已归档模块的代码实现、API参数、修Bug或具体配置
- **强制动作**：在生成代码/回复前，**必须显式调用文件读取工具** (如 `read_file`, `cat`) 读取对应的 `docs/*.md` 文件
- **禁止脑补**：绝对不允许根据文件名猜测内容。若未读取文件直接输出细节代码，视为严重违规
